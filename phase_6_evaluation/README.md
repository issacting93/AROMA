# Phase 6 — Evaluation & Analysis (weeks 22–24)

## Purpose

Synthesise all findings from the pipeline evaluation (Phase 5) into a coherent analysis. This is where you step back from the method and ask: **what did we learn about care roles in AI mental health systems?**

---

## Steps

### 6.1 Coverage Gap Analysis

Using the [Taxonomy Cross-Reference Map](../taxonomy-table.html) and the coded data:

| Question | Analysis |
|----------|----------|
| Which AROMA dimensions are well-covered by existing literature? | D1 (Support Type) — already theorised by Cutrona & Suhr |
| Which are under-theorised? | D2 (Care Role), D3 (Support Strategy) |
| Which role–dimension combinations are most empirically supported? | Frequency counts from coded data |
| Which are theoretically predicted but empirically absent? | Roles the codebook defines but the data rarely instantiates |

### 6.2 Role Distribution Analysis

From the gold standard coded data:
- Role frequency distribution across datasets (WildChat vs. Reddit vs. others)
- Role co-occurrence and transition patterns (from turn-level data)
- Relationship between user prompt type and enacted care role
- Authority-Agency Paradox frequency and patterns

### 6.3 Failure Mode Taxonomy

From Phase 5 error analysis, formalise:
- Most common failure modes per role
- Failure modes that are **structurally unavoidable** (the AI enacts Advisor authority without agency)
- Failure modes that are **designable** (system prompt engineering can prevent)
- Design recommendations per failure mode

### 6.4 Contribution Framing

Articulate three levels of contribution:
1. **Taxonomy contribution** — AROMA as a multi-dimensional framework
2. **Empirical contribution** — first systematic coding of care roles in AI mental health conversations
3. **Design contribution** — failure mode taxonomy with actionable design implications

---

- [ ] Failure mode taxonomy (categorised, with examples)
- [ ] Design implications document (what system designers should do differently)
